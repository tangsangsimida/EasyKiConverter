    ADR 001: 选择 MVVM 架构¶

    状态¶

    已接受

    上下文¶

    EasyKiConverter 是一个基于 Qt 6 Quick 的桌面应用程序，用于将 EasyEDA 元件转换为 KiCad 格式。在项目初期，我们使用的是简单的 MVC（Model-View-Controller）架构，但随着项目的发展，我们遇到了以下问题：

    1. 代码耦合度高：Controller 层承担了太多职责，包括业务逻辑、UI 状态管理和数据转换，导致代码难以维护和测试。

    2. UI 状态管理混乱：UI 状态分散在 Controller 和 View 中，难以追踪和同步。

    3. 测试困难：由于业务逻辑和 UI 代码耦合，单元测试难以编写和维护。

    4. 可扩展性差：添加新功能需要修改多个地方，容易引入错误。

    5. 代码复用性低：相似的业务逻辑在不同地方重复实现。

    我们需要一个更好的架构来解决这些问题。

    决策¶

    我们选择采用 MVVM（Model-View-ViewModel）架构，理由如下：

    架构设计¶

    层次职责¶

    1. Model 层
    2. 负责数据存储和管理
    3. 不包含任何业务逻辑

    4. 纯粹的数据模型

    5. Service 层

    6. 负责业务逻辑处理
    7. 提供核心功能

    8. 调用底层 API

    9. ViewModel 层

    10. 管理 UI 状态
    11. 处理用户输入
    12. 调用 Service 层

    13. 数据绑定和转换

    14. View 层

    15. 负责 UI 显示和用户交互
    16. 使用 QML 实现
    17. 通过数据绑定与 ViewModel 通信

    选择理由¶

    1. 清晰的关注点分离
    2. Model 只负责数据
    3. ViewModel 只负责 UI 状态和业务逻辑调用
    4. View 只负责 UI 显示

    5. Service 只负责业务逻辑

    6. 更好的可测试性

    7. ViewModel 可以独立于 View 进行测试
    8. Service 层可以独立进行单元测试

    9. Model 层易于测试

    10. 更好的可维护性

    11. 代码组织清晰
    12. 职责明确

    13. 易于定位和修复问题

    14. 更好的可扩展性

    15. 添加新功能只需要在相应的层添加代码
    16. 不影响其他层

    17. 易于添加新的 ViewModel

    18. 与 Qt Quick 的天然契合

    19. Qt Quick 的数据绑定机制与 MVVM 完美匹配
    20. QML 作为 View 层非常合适

    21. Qt 的信号槽机制实现了观察者模式

    22. 更好的团队协作

    23. 不同开发者可以专注于不同的层
    24. UI 开发者专注于 QML
    25. 逻辑开发者专注于 C++
    26. 减少冲突

    2. 代码耦合度降低
    3. 代码复用性提高

    4. 代码可读性增强

    5. 开发效率提高

    6. 新功能开发更快速
    7. Bug 修复更容易

    8. 代码审查更高效

    9. 测试覆盖度提高

    10. 单元测试更容易编写
    11. 测试覆盖率从 30% 提升到 80%+

    12. 集成测试更完善

    13. 维护成本降低

    14. 问题定位更快速
    15. 修改影响范围更小

    16. 重构更安全

    17. 用户体验改善

    18. UI 响应更快
    19. 状态管理更稳定
    20. 错误处理更完善

    消极后果¶

    1. 学习曲线
    2. 新开发者需要理解 MVVM 架构

    3. 需要额外的培训和文档

    4. 初始开发成本

    5. 架构重构需要时间
    6. 需要编写更多的代码

    7. 需要建立新的开发流程

    8. 过度设计的风险

    9. 对于简单的功能，MVVM 可能过于复杂
    10. 需要权衡架构复杂度

    缓解措施¶

    1. 完善文档
    2. 提供详细的架构文档
    3. 提供开发指南

    4. 提供示例代码

    5. 渐进式迁移

    6. 不是一次性重构所有代码
    7. 逐步迁移各个模块

    8. 保持系统稳定

    9. 代码审查

    10. 确保新代码遵循 MVVM 架构
    11. 定期审查代码质量
    12. 及时纠正偏离

    替代方案¶

    1. 保持 MVC 架构¶

    优点： - 不需要重构 - 团队熟悉

    缺点： - 无法解决现有问题 - 维护成本持续增加

    2. 使用 MVP 架构¶

    优点： - 比 MVC 更清晰的分离 - View 和 Model 完全解耦

    缺点： - Presenter 仍然可能变得复杂 - 与 Qt Quick 的契合度不如 MVVM

    3. 使用 Redux 模式¶

    优点： - 状态管理非常清晰 - 易于调试

    缺点： - 对于 Qt Quick 来说过于复杂 - 学习曲线陡峭

    结论¶

    经过评估，我们选择 MVVM 架构作为 EasyKiConverter 的架构模式。MVVM 架构提供了清晰的关注点分离、更好的可测试性和可维护性，与 Qt Quick 的数据绑定机制完美契合。虽然初始迁移成本较高，但长期来看，这将大大提高开发效率和代码质量。
